Data fetching in the App Router is built around a simple idea: your Server Components are just async functions, so you can `await` data at the top of the component tree without any extra libraries or boilerplate. Next.js extends the native `fetch` API with powerful caching controls that let you choose exactly how fresh your data needs to be.

## The Simplest Pattern

The most straightforward way to fetch data is to define an async helper function and call it inside an async Server Component. Because this all happens on the server, the browser never sees the fetching logic — only the finished HTML.

```tsx
// app/posts/page.tsx
async function getPosts() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts')
  if (!res.ok) throw new Error('Failed to fetch posts')
  return res.json()
}

export default async function PostsPage() {
  const posts = await getPosts()
  return (
    <ul>
      {posts.slice(0, 5).map((post: { id: number; title: string }) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

<Note>
  If `fetch` throws or returns a non-OK response, the error bubbles up to the nearest `error.tsx` boundary. Always validate `res.ok` before calling `res.json()`.
</Note>

## Caching Strategies

Next.js caches `fetch` responses by default. You control caching behaviour through the second argument to `fetch`. Pick the strategy that matches how often your data changes.

<Steps>
  <Step title="Default — Cached Forever (Static)">
    When you call `fetch` with no extra options, Next.js caches the response indefinitely and serves it from the cache on every request. This is equivalent to `{ cache: 'force-cache' }`.

    ```tsx
    // app/products/page.tsx
    async function getProducts() {
      const res = await fetch('https://api.example.com/products')
      // No options — cached at build time and reused forever
      return res.json()
    }
    ```

    Use this for data that never changes, like a list of countries or a static product catalogue.
  </Step>

  <Step title="No Cache — Always Fresh (Dynamic)">
    Pass `{ cache: 'no-store' }` to opt out of caching entirely. Next.js fetches fresh data on every single request, making the page fully dynamic.

    ```tsx
    // app/feed/page.tsx
    async function getFeed() {
      const res = await fetch('https://api.example.com/feed', {
        cache: 'no-store',
      })
      return res.json()
    }
    ```

    Use this for data that changes constantly — user dashboards, live scores, or notification feeds.
  </Step>

  <Step title="Revalidate — Time-Based ISR">
    Pass `{ next: { revalidate: 60 } }` to use Incremental Static Regeneration (ISR). Next.js serves the cached response but regenerates it in the background once the time-to-live (in seconds) expires.

    ```tsx
    // app/blog/page.tsx
    async function getPosts() {
      const res = await fetch('https://api.example.com/posts', {
        next: { revalidate: 60 }, // Regenerate at most once per minute
      })
      return res.json()
    }
    ```

    Use this for data that changes occasionally — blog posts, product prices, or documentation pages.
  </Step>
</Steps>

## Static vs Dynamic Rendering

The caching strategy you choose directly determines whether Next.js pre-renders the page or renders it on demand.

<CardGroup cols={2}>
  <Card title="Static Rendering" icon="bolt" href="https://nextjs.org/docs/app/building-your-application/rendering/server-components#static-rendering-default">
    Pages that use cached `fetch` calls (or no fetching at all) are rendered at **build time**. Next.js generates the HTML once and reuses it for every visitor. This is the fastest possible option because the response comes straight from a CDN edge.
  </Card>
  <Card title="Dynamic Rendering" icon="arrows-rotate" href="https://nextjs.org/docs/app/building-your-application/rendering/server-components#dynamic-rendering">
    Pages that use `{ cache: 'no-store' }`, read incoming request data (headers, cookies, search params), or use dynamic APIs like `cookies()` are rendered **on every request**. The HTML is generated fresh each time on the server.
  </Card>
</CardGroup>

<Tip>
  You don't have to choose one strategy for an entire app. Different pages — even different `fetch` calls on the same page — can each have their own caching strategy. Mix and match freely.
</Tip>

## Quick Reference

| Option | Rendered | When to Use |
|---|---|---|
| `fetch(url)` | Build time | Truly static data |
| `{ next: { revalidate: N } }` | Periodically | Occasionally updated content |
| `{ cache: 'no-store' }` | Every request | Always-fresh, user-specific data |
