Before you write your first line of Next.js code, it helps to understand why the framework exists, what problem it solves, and why so many teams — from scrappy startups to household-name companies — have standardised on it. This context will make everything that follows feel less arbitrary and more intentional.

## The Problem With Plain React

React is a UI library, not a framework. That is a feature, not a flaw — React gives you enormous flexibility. But that flexibility comes with a cost: **you have to make a lot of decisions yourself**.

When you build with plain React (or Create React App), you quickly run into questions the library does not answer for you:

- How do you handle routing between pages?
- How do you fetch data on the server before rendering?
- How do you optimise images for different screen sizes?
- Where do you put your API logic?
- How do you make your app fast enough for real users?

Next.js answers all of these questions with well-designed, production-tested conventions. You spend less time configuring infrastructure and more time building features.

## The App Router: The Modern Way to Build

Next.js has two routing systems. The older one — the **Pages Router** — is well-established and still widely used. The newer one — the **App Router** — was introduced in Next.js 13 and became fully stable in Next.js 14.

This course teaches the **App Router exclusively**, and for good reason:

- It is the officially recommended approach going forward.
- It introduces **React Server Components**, which let you fetch data directly inside your UI components without extra API calls.
- It supports **nested layouts**, making it easy to share UI elements like sidebars and headers across multiple pages without duplicating code.
- It aligns closely with the direction React itself is heading, so skills transfer.

<Note>
  You will occasionally encounter tutorials or Stack Overflow answers that use the Pages Router. The patterns look slightly different. When in doubt, check this handbook — all examples here use the App Router.
</Note>

## Who Uses Next.js in Production

Next.js is not an experimental technology. It is battle-tested at massive scale:

- **Vercel** — the company behind Next.js, running their own platform on it.
- **Hulu** — serving millions of streaming users with Next.js-powered pages.
- **TikTok** — using Next.js for parts of their web experience.
- **OpenAI** — the ChatGPT web interface is built with Next.js.
- **Thousands of startups** — because it offers a fast path from idea to deployed product.

This adoption means a thriving ecosystem: abundant tutorials, active community forums, plentiful job postings, and a framework that will continue to be maintained and improved.

## Full-Stack in One Framework

One of the most compelling things about Next.js is that it collapses the traditional separation between frontend and backend into a single, cohesive codebase.

```
my-app/
├── app/
│   ├── page.tsx          ← Your homepage (frontend)
│   ├── about/
│   │   └── page.tsx      ← /about route (frontend)
│   └── api/
│       └── users/
│           └── route.ts  ← /api/users endpoint (backend)
```

Your React components and your API endpoints live side by side. You can share TypeScript types, utility functions, and database logic between them without maintaining two separate repositories. For individuals and small teams especially, this is a massive productivity advantage.

<Tip>
  If you have ever had to coordinate a separate Express or FastAPI backend with a React frontend, you will immediately appreciate how much simpler the Next.js full-stack model feels.
</Tip>
