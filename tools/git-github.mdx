Version control is the safety net that lets you experiment freely, collaborate without fear, and ship confidently. Git records every change you make to your project so you can travel back in time, see exactly what changed, and work alongside teammates without stepping on each other's code. If you're building with Next.js, Git and GitHub aren't optional extras — they're part of your everyday workflow from day one.

## Installing Git

Download Git from [git-scm.com](https://git-scm.com). The installer works on macOS, Windows, and Linux. Once it's installed, verify it's working by running:

```bash
git --version
```

You should see something like `git version 2.44.0`. Any version above 2.30 is fine for this course.

<Note>
  macOS users may already have Git installed via Xcode Command Line Tools. If the version check works, you're good to go.
</Note>

## Essential Git Commands for Next.js Projects

These six commands cover 90% of what you'll do on a daily basis. Open your terminal inside your Next.js project folder and try them:

```bash
git init                              # initialize a new repo in the current folder
git add .                             # stage all changed files for the next commit
git commit -m "feat: add homepage"    # save a snapshot with a descriptive message
git status                            # see which files have changed since last commit
git log --oneline                     # view a compact history of your commits
git diff                              # see exactly what lines changed since last commit
```

Here's the cycle you'll repeat constantly: make a change, run `git add .` to stage it, then `git commit` to save the snapshot. That's it.

## Setting Up GitHub

GitHub is a cloud host for your Git repositories. It's where your code lives remotely, gets deployed from, and can be shared with collaborators.

<Steps>
  <Step title="Create a GitHub Account">
    Go to [github.com](https://github.com) and sign up for a free account. Choose a username you're happy to put on a CV — it'll appear in your repository URLs.
  </Step>
  <Step title="Create a New Repository">
    Click the **+** icon in the top-right corner and select **New repository**. Give it a name (e.g. `my-first-app`), leave it set to **Public**, and click **Create repository**. Don't initialize it with a README — you already have local code.
  </Step>
  <Step title="Connect Your Local Project to GitHub">
    GitHub shows you the exact commands to run. In your project terminal, run:

    ```bash
    git remote add origin https://github.com/your-username/my-first-app.git
    git branch -M main
    git push -u origin main
    ```

    After this, refresh your GitHub page — your code is now live in the cloud.
  </Step>
</Steps>

## The `.gitignore` File

When `create-next-app` scaffolds your project, it automatically creates a `.gitignore` file. This file tells Git which files and folders to **never** track. You don't need to edit it, but you should know what's in there:

```bash
# .gitignore (auto-generated by create-next-app)
node_modules/     # thousands of dependency files — never commit these
.next/            # build output — regenerated on each build
.env.local        # secret keys and environment variables — never commit these
*.log             # log files
```

The most important rule: **never commit `.env.local`**. It contains API keys and secrets that must stay off GitHub.

## Writing Good Commit Messages

A commit message should tell your future self — or a teammate — *why* a change was made, not just what files changed. The most widely used format is **Conventional Commits**:

| Prefix | When to use it |
|--------|----------------|
| `feat:` | Adding a new feature |
| `fix:` | Fixing a bug |
| `chore:` | Maintenance tasks (updating deps, config changes) |
| `style:` | Formatting only — no logic changes |
| `docs:` | Documentation updates |
| `refactor:` | Restructuring code without changing behavior |

Examples:

```bash
git commit -m "feat: add contact form to homepage"
git commit -m "fix: resolve hydration error in navbar"
git commit -m "chore: update next to 14.2.0"
```

<Tip>
  Commit often — small, focused commits are far better than one giant commit at the end of the day. If your commit message needs the word "and" to describe everything it does, it should probably be two commits. Small commits are easier to review, easier to revert, and easier to understand six months later.
</Tip>
